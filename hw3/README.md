# 중단 후 재개 가능한 TCP 파일 업로더

TCP 소켓으로 파일을 업로드하는 C 기반 클라이언트/서버 프로그램입니다. 업로드 도중 연결이 끊기더라도 서버에 저장된 파일 크기를 기준으로 전송을 이어서 수행할 수 있으며, 서버는 여러 클라이언트를 스레드로 동시에 처리합니다.

## 주요 기능

- 파일을 1,024바이트 단위로 전송
- 각 청크가 저장될 때마다 서버가 누적 저장 바이트 수를 ACK로 응답
- 연결이 끊어지면 클라이언트가 1초 간격으로 재접속 후 마지막 저장 위치부터 전송 재개
- 서버 또는 클라이언트가 재시작되어도 기존 파일 크기를 이용해 업로드 재개
- 클라이언트 ID별 디렉터리에 업로드 파일 저장
- 동시에 접속한 클라이언트 ID의 중복 사용 거부
- POSIX 스레드를 이용한 다중 클라이언트 처리

## 파일 구성

| 파일 | 설명 |
| --- | --- |
| `hw3_server.c` | 접속 처리, 파일 저장, ACK 전송을 담당하는 멀티스레드 서버 |
| `hw3_client.c` | 파일 전송 및 연결 복구를 담당하는 클라이언트 |
| `Makefile` | 서버와 클라이언트 빌드 규칙 |
| `hw02_22200376(Final Upload).pdf` | 메시지 형식, 흐름도, 오류 처리 설명 문서 |

## 요구 환경

- Linux 또는 macOS 등 POSIX 호환 환경
- GCC 또는 호환 C 컴파일러
- `make`
- POSIX Threads (`pthread`)

## 빌드

```bash
cd hw3
make
```

빌드가 완료되면 다음 실행 파일이 생성됩니다.

- `hw3_server`
- `hw3_client`

생성된 실행 파일을 삭제하려면 다음 명령을 사용합니다.

```bash
make clean
```

## 실행 방법

먼저 서버를 실행합니다.

```bash
./hw3_server <port>
```

예를 들어 9000번 포트를 사용하려면:

```bash
./hw3_server 9000
```

다른 터미널에서 클라이언트를 실행합니다.

```bash
./hw3_client <server_ip> <port> <client_id> <file_name>
```

서버와 클라이언트를 같은 컴퓨터에서 실행하며 `sample.txt`를 전송하는 예시는 다음과 같습니다.

```bash
./hw3_client 127.0.0.1 9000 client01 sample.txt
```

클라이언트는 현재 작업 디렉터리에 있는 `sample.txt`를 읽습니다. 서버는 서버 실행 디렉터리를 기준으로 아래 위치에 파일을 저장합니다.

```text
client01/sample.txt
```

따라서 서버와 클라이언트를 같은 디렉터리에서 실행하면 원본과 저장 대상의 경로가 겹치지 않도록 서로 다른 클라이언트 ID를 사용해야 합니다.

## 동작 방식

1. 클라이언트가 서버에 접속해 클라이언트 ID와 파일 이름을 전송합니다.
2. 서버는 `<client_id>/<file_name>`의 기존 크기(`saved_bytes`)를 클라이언트에 전달합니다.
3. 클라이언트는 원본 파일에서 `saved_bytes` 위치로 이동합니다.
4. 클라이언트가 파일 청크를 전송하면 서버는 저장 후 누적 바이트 수를 ACK로 반환합니다.
5. 연결이 끊어지면 클라이언트가 재접속하고, 서버가 새로 알려준 파일 크기부터 전송을 재개합니다.
6. 전송이 완료되면 클라이언트가 쓰기 방향을 종료하고 서버가 최종 저장 크기를 전송합니다.

이미 서버에 저장된 파일 크기가 원본 파일 크기와 같으면 클라이언트는 업로드가 완료된 것으로 판단하고 종료합니다.

## 업로드 재개 확인

큰 파일을 업로드하는 도중 서버 또는 클라이언트를 `Ctrl+C`로 종료한 다음 같은 인자로 다시 실행하면, 서버에 남아 있는 파일 크기 이후부터 업로드가 계속됩니다.

전송 결과는 다음과 같이 비교할 수 있습니다.

```bash
cmp sample.txt client01/sample.txt
```

출력이 없고 종료 코드가 0이면 두 파일이 같습니다.

## 현재 구현의 제약사항

- 클라이언트 ID와 파일 이름은 고정 크기 버퍼를 사용하므로 각각 31자 이하의 단순한 이름을 사용해야 합니다.
- 파일 이름에 디렉터리 경로를 포함하면 저장 경로 생성이나 재개 동작이 의도대로 작동하지 않을 수 있습니다.
- 같은 클라이언트 ID는 동시에 한 연결만 사용할 수 있습니다. 중복 접속은 `ERROR_DUPLICATE`로 거부됩니다.
- 서버는 기존 파일의 크기만으로 재개 위치를 판단하므로, 같은 클라이언트 ID와 파일 이름으로 내용이 다른 파일을 전송하지 않아야 합니다.
- 서버에 저장된 파일이 원본보다 큰 경우를 별도로 복구하지 않습니다.
- 정수 메시지는 네트워크 바이트 순서로 변환하거나 직렬화하지 않은 네이티브 `long` 형식이므로, 서로 다른 아키텍처의 시스템 간 호환은 보장되지 않습니다.
- 인증, 암호화, 파일 무결성 해시 검증은 구현되어 있지 않으므로 신뢰할 수 있는 실습 환경에서 사용하는 것을 전제로 합니다.

## 참고

이 프로젝트는 네트워크 단절, 서버/클라이언트 재시작, ACK 수신 실패 상황에서 서버가 보유한 `saved_bytes`를 기준으로 파일 업로드를 복구하는 과제 구현입니다.
